# E2E テストケース目録

## 1. 目的と前提

- **目的:** 本アプリの主要機能について、E2E（ブラウザ相当）の観点で検証すべき挙動を、タイトル・前提（Given）・操作（When）・期待（Then）の形式で列挙する。
- **自動 E2E の現状:** 本リポジトリには、Playwright / Cypress 等の E2E 自動化スクリプトは **未整備** である。本ドキュメントは、手動テストおよび将来自動化の設計用の **テスト仕様** である。
- **画面構成の注意:** 旧仕様（`docs/spec.md`）に `/result` があるが、**現行実装はトップ（`/`）1 画面内に生成結果と履歴を表示** する。以下のケースは実装に合わせる。

## 3. 認証・ルーティング

### TC-AUTH-01

- **タイトル:** 未認証ではホームに入れない
- **前提（Given）:** ブラウザに `Supabase` の有効なセッションがない
- **操作（When）:** `/`（ホーム）に直接アクセスする
- **期待（Then）:** `/login` にリダイレクトされる

### TC-AUTH-02

- **タイトル:** 正しいパスワードでログインできる
- **前提（Given）:** `/login` を開き、有効なアカウントのメール・パスワードを入力可能である
- **操作（When）:** 「ログイン」ボタンを押す
- **期待（Then）:** 「ログイン成功」のアラートが表示され、その後 `/`（ホーム）に遷移する

### TC-AUTH-03

- **タイトル:** 誤ったパスワードではログインできない
- **前提（Given）:** `/login` を開いている
- **操作（When）:** 存在するメールに誤ったパスワードを入れ「ログイン」を押す
- **期待（Then）:** 「ログイン失敗: …」の形式でアラートが表示され、ホームには遷移しない

### TC-AUTH-04

- **タイトル:** 新規登録を試みたときのフィードバック
- **前提（Given）:** `/login` を開いている
- **操作（When）:** メールとパスワードを入力し「新規登録」を押す
- **期待（Then）:** エラー時は「登録失敗: …」のアラート。成功時は「登録処理を実行しました。ログインしてください（…）」のアラート（実装の文言通り）

### TC-AUTH-05

- **タイトル:** ログアウトでセッションが切れる
- **前提（Given）:** ホーム（`/`）にログイン済みで表示している
- **操作（When）:** 「ログアウト」ボタンを押す
- **期待（Then）:** `/login` に遷移し、以降 `/` 直打ちであれば再びログインへ誘導される

### TC-AUTH-06

- **タイトル:** ログイン済みでログイン画面に来た場合の挙動
- **前提（Given）:** 既にセッションが有効である
- **操作（When）:** `/login` を開く（手動で URL 入力）
- **期待（Then）:** アプリの仕様としてはログイン画面が表示される（自動で `/` に戻す実装は任意。回帰時に仕様化してよい）

## 4. メイン画面（補助テキスト・AI 生成）

### TC-GEN-01

- **タイトル:** 補助テキストのみで商品情報を生成できる
- **前提（Given）:** ホームにログイン済みで、画像は未選択である
- **操作（When）:** 補助情報テキストを入力し「生成する」を押す
- **期待（Then）:** しばらく「生成中...」表示のあと、見出し・説明・ハッシュタグ・カテゴリ・ブランド・価格（円）が画面に表示される

### TC-GEN-02

- **タイトル:** 生成中は連打を防ぐ
- **前提（Given）:** ホームにログイン済みである
- **操作（When）:** 「生成する」を押し、完了する前にボタンの状態を確認する
- **期待（Then）:** ボタン文言が「生成中...」となり、操作できない（disabled）状態である

### TC-GEN-03

- **タイトル:** 生成前は直前の生成結果表示が消える
- **前提（Given）:** 既に1回生成済みで結果ブロックが表示されている
- **操作（When）:** 再度「生成する」を押す
- **期待（Then）:** 新しいリクエスト開始時点で、直前の結果表示が消えてから（`setResult(null)`）ローディングに入る

### TC-GEN-04

- **タイトル:** 画像付きで Vision 推定の生成ができる
- **前提（Given）:** ホームにログイン済み、1 枚以上の画像を添付済みである
- **操作（When）:** 任意の補助テキストとともに「生成する」を押す
- **期待（Then）:** API が成功し、同様に商品情報が表示され、失敗しなければ履歴にも追加される

## 5. 画像（枚数・プレビュー・圧縮）

### TC-IMG-01

- **タイトル:** 選択画像のプレビューが出る
- **前提（Given）:** ホームにログイン済みである
- **操作（When）:** ファイル入力で 1 枚以上の画像を選ぶ
- **期待（Then）:** 幅 100px 程度のサムネイルが横に並び表示される

### TC-IMG-02

- **タイトル:** 5 枚まで採用される
- **前提（Given）:** ホームにログイン済み、6 枚以上入ったフォルダや複数枚選択が可能な環境である
- **操作（When）:** 6 枚以上を一度に選択する
- **期待（Then）:** プレビ枚数は **5 枚まで**（先頭 5 枚のみ使用）に制限される

### TC-IMG-03

- **タイトル:** 圧縮後データが生成 API に送られる
- **前提（Given）:** 画像を 1 枚以上選択し「生成する」まで行う
- **操作（When）:** ネットワークのリクエスト本文を確認できる（DevTools 等）
- **期待（Then）:** `images` 配列に `data:` URL 形式（JPEG 圧縮後）の文字列が含まれる

## 6. 履歴（一覧・空状態・更新）

### TC-HIST-01

- **タイトル:** 履歴が空のメッセージ
- **前提（Given）:** ログイン済みで、当該ユーザーに `products` の行がまだない（初回等）
- **操作（When）:** ホームの「履歴」セクションを見る
- **期待（Then）:** 「履歴がありません」と表示される

### TC-HIST-02

- **タイトル:** 生成成功後、履歴に 1 件追加される
- **前提（Given）:** ホームにログイン済み、履歴は空または既存行がある
- **操作（When）:** 正常に 1 回「生成する」が完走する
- **期待（Then）:** 「履歴」に新しいカードが先頭付近に現れ、タイトル・説明・各種フィールド・サムネイル画像（あれば）が表示される

### TC-HIST-03

- **タイトル:** 履歴行に操作ボタンがある
- **前提（Given）:** 履歴に 1 件以上ある
- **操作（When）:** 当該行を見る
- **期待（Then）:** 「編集」「削除」「コピー」が表示される

## 7. 履歴の編集・削除・コピー

### TC-CRUD-01

- **タイトル:** 編集モーダルでタイトル等を更新できる
- **前提（Given）:** 履歴に 1 件以上ある
- **操作（When）:** 「編集」を押し、タイトル・説明・タグ欄を変更して「保存」を押す
- **期待（Then）:** モーダルが閉じ、一覧上の同じ行の内容が更新されて表示される

### TC-CRUD-02

- **タイトル:** タグはスペース区切りで配列化される
- **前提（Given）:** 編集モーダルを開いている
- **操作（When）:** タグ欄に `foo bar baz` のようにスペース区切りで入力し「保存」する
- **期待（Then）:** 表示上 3 つのタグとして扱われる（スペースで split）

### TC-CRUD-03

- **タイトル:** 編集をキャンセルできる
- **前提（Given）:** 編集モーダルを開いている
- **操作（When）:** 内容を変え「キャンセル」を押す
- **期待（Then）:** モーダルが閉じ、一覧の元データに変更は反映されない

### TC-CRUD-04

- **タイトル:** 履歴行を削除できる
- **前提（Given）:** 履歴に対象行がある
- **操作（When）:** 「削除」を押す
- **期待（Then）:** 当該行が一覧から消える

### TC-CRUD-05

- **タイトル:** 履歴からコピーできる
- **前提（Given）:** 履歴に 1 件以上ある。ブラウザがクリップボードに書き込み可能（権限・HTTPS 等）
- **操作（When）:** 「コピー」を押す
- **期待（Then）:** 「コピーしました」アラートが出る。クリップボードにはタイトル・説明・タグを含む所定のテキスト形式が入る  
  - **E2E 自動化の注意:** Playwright 等では `clipboard-read` 権限付与、または `navigator.clipboard.readText` をテスト専用フックで代替する方針を文書化してよい

## 8. API・エラー（直接または UI 経由）

### TC-API-01

- **タイトル:** 認証ヘッダなしの POST は 401
- **前提（Given）:** `POST /api/generate` をクライアント（curl / REST クライアント等）で呼べる
- **操作（When）:** `Authorization` ヘッダを付けずに JSON ボディを送る
- **期待（Then）:** HTTP 401 と `Unauthorized` 相当の JSON

### TC-API-02

- **タイトル:** 無効トークンでは 401
- **前提（Given）:** 同様に API を直接叩ける
- **操作（When）:** 不正な `Bearer` トークンを付与して POST する
- **期待（Then）:** HTTP 401

### TC-API-03

- **タイトル:** 正常系で GET ヘルスが返る
- **操作（When）:** `GET /api/generate` を呼ぶ
- **期待（Then）:** 本文に `API is working` が含まれる

### TC-ERR-01

- **タイトル:** 生成 API がエラーを返したときの UI
- **前提（Given）:** ネットワークで `/api/generate` が 4xx/5xx を返す状況（API キー未設定、AI 失敗、JSON パース失敗等）
- **操作（When）:** 通常どおり「生成する」を押す
- **期待（Then）:** **現行フロントは** レスポンスを `res.json()` のみしており、ユーザー向けのエラーメッセージ表示は未実装。E2E では「新しい成功結果が表示されない」「コンソールに失敗が出る」等の観測に留まる。仕様上はエラー表示を求めるなら、別タスクで UI を足したうえで本ケースを更新する

## 9. 非機能（目安・注意）

### TC-NF-01

- **タイトル:** 生成待ち時間の目安
- **前提（Given）:** README・仕様上「5〜15 秒程度」を目安としている
- **操作（When）:** 正常系で「生成する」を押し、表示が揃うまで待つ
- **期待（Then）:** E2E のタイムアウトは余裕を持たせる（厳密な上限制約は、ステージング負荷に依存するため **手動または緩い閾値** 推奨）

### TC-NF-02

- **タイトル:** 対象ブラウザ
- **前提（Given）:** 仕様上 PC 利用を主とする（`docs/spec.md` の非機能）
- **操作（When）:** 本番同様のデスクトップブラウザで上記を実施
- **期待（Then）:** 主要ブラウザ（例: Chrome 最新）で上記主要フローが通る
